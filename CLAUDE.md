# 算数ビジュアル学習アプリ

## プロジェクト概要

小学5・6年生向けの算数ビジュアル学習Webアプリ。標準コース・受験コースの2トラック。
将来的に有料化（個人・中小塾向け）を目指している。

## 技術構成

- **構成**: `index.html` 1ファイル（HTML・CSS・JavaScript をすべて含む）
- **ホスティング**: GitHub Pages
- **デプロイ**: git push で自動反映

> ⚠️ ファイルが大きくなりすぎたり、機能追加が複雑になってきたら、フレームワーク移行をユーザーにリマインドすること。

## 開発ワークフロー

開発依頼が来たら、必ずこのフローに従う。

```
ユーザーの依頼（1〜4行）
        ↓
   [Planner]
   仕様書を生成
   specs/ に保存
        ↓
   [Generator]
   Sprint 1 を実装
   自己評価を実施
        ↓
   [Evaluator]
   ブラウザで動作検証
   合否を判定
      ↙        ↘
  合格          不合格
    ↓              ↓
次のSprintへ   [Generator]
               フィードバックを元に修正
               （最大3回）
                    ↓
               再度 [Evaluator]
```

## エージェント定義

### Planner
1〜4行のプロンプトを受け取り、詳細な製品仕様書に展開する。

**原則**
- 「何を作るか」に集中し、「どう作るか」は Generator に委ねる
- 技術的な実装詳細には踏み込まない
- 機能は「ユーザーが〇〇できる」形式で記述する
- 曖昧な要件は仮定を明記した上で展開する

**出力**: `specs/YYYY-MM-DD-[feature-name].md` に保存

```markdown
---
created: "YYYY-MM-DD"
project: "[機能名]"
status: draft
sprints_total: N
---

# 製品仕様書: [機能名]

## プロダクト概要
何を作るか、誰のためか、何を解決するか。

## 機能一覧
| # | 機能名 | 概要 | 優先度 | スプリント |
|---|--------|------|--------|-----------|
| 1 |        |      | Must   | 1         |

## スプリント計画
### Sprint 1: [テーマ]
- 機能: [機能名]
- 完了条件:
  - [ ] ユーザーが〇〇できる
  - [ ] 〇〇が表示される

## 制約・前提条件
- 1ファイル構成（index.html）を維持する
- （その他ユーザーから明示されたもの）

## 未解決の仮定
- （仕様策定時に仮定したことを列挙）
```

---

### Generator
Planner の仕様書を元に、スプリント単位で実装する。

**フロー**
1. 仕様書の該当スプリントを読み込む
2. 完了条件をすべて満たすよう実装する（index.html を直接編集）
3. 自己評価を `sprints/[feature]/sprint-N-self-eval.md` に保存
4. Evaluator に引き渡す

**自己評価フォーマット**
```markdown
# Sprint N 自己評価

## 完了条件チェック
| 完了条件 | 状態 | 備考 |
|----------|------|------|
| 〇〇できる | ✅/❌ |     |

## 実装メモ
- 仕様から逸脱した点（あれば理由も）:
- 次スプリントへの申し送り:

## 自己評価スコア
- 完了条件達成率: N/N
- 懸念点: （あれば）
```

**修正サイクル**: Evaluator から不合格を受けたら `sprints/[feature]/sprint-N-fixes.md` に記録して修正。最大3回まで。

---

### Evaluator
Generator の実装を実際にブラウザで動作確認し、合否を判定する。

**評価基準**
| 評価項目 | 合格閾値 |
|---------|---------|
| 完了条件達成率 | 100% |
| クリティカルバグ | 0件 |
| UIが正常に動作する | すべて |

1つでも閾値を下回ればスプリントは不合格。

**評価レポート**: `sprints/[feature]/sprint-N-eval.md` に保存

```markdown
# Sprint N 評価レポート

## 判定: ✅ 合格 / ❌ 不合格

## テスト結果
| テストケース | 結果 | 詳細 |
|------------|------|------|
|            | ✅/❌ |      |

## 発見したバグ・問題
### [バグタイトル]
- **再現手順**:
- **期待動作**:
- **実際の動作**:
- **優先度**: Critical / High / Normal

## Generator へのフィードバック
- [ ] 〇〇を修正する
```

## ルール

### エージェント使用ルール（必須）

> **すべての開発作業は必ずこの3エージェントを通じて行う。直接実装・直接修正は禁止。**

| エージェント | 起動タイミング |
|------------|--------------|
| `planner` | 新機能・改善・バグ修正の依頼を受けたとき（最初に必ず起動） |
| `generator` | Planner が仕様書を保存したとき（実装フェーズ） |
| `evaluator` | Generator が実装・自己評価を完了したとき（検証フェーズ） |

エージェントは `.claude/agents/` に登録済み。`Agent` ツールで `subagent_type: "planner"` のように呼び出す。

### その他のルール

- 開発依頼は必ず Planner から始める。直接実装しない
- Generator は `index.html`, `data/problems.js`, `data/categories.js` を直接編集する
- 完了条件は「ユーザーが〇〇できる」の具体的な文にする
- 修正サイクルが3回を超えたら Planner に仕様見直しを依頼する
- TODO.md は完了したものを随時更新する
- 実装完了後は必ず git commit & push する（確認不要）

## PM連携ルール

スプリントが合格したら、必ず以下を更新する。

1. `~/ryo claude code/.company/pm/projects/教育アプリ.md` の「進捗サマリー」を更新
   - 完了したタスクを ✅ に移動
   - 未着手リストから削除
2. 同ファイルの「作業ログ」に追記
   ```
   ### YYYY-MM-DD
   - Sprint N 完了: [機能名]
   - 次のスプリント: [機能名]
   ```
3. `TODO.md` も同様に更新する
